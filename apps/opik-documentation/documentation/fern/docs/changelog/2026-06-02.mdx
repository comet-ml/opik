## Prompt Environments for the Prompt Library

Prompt versions can now be tagged with deployment environments (e.g. `production`, `staging`) and retrieved by environment name from both the Python and TypeScript SDKs. This makes it straightforward to point your agent at the right prompt version per deployment stage without hard-coding version hashes.

**What's new:**

- **Fetch by environment** ‚Äî `client.get_prompt(name, environment="production")` returns the version currently tagged as production; `version` and `environment` are mutually exclusive and passing both raises a clear error
- **Assign environments** ‚Äî `client.set_prompt_environments(name, ["production", "staging"])` replaces the full environment set on a version; the same environment is automatically moved away from whatever version previously held it
- **Tag at creation** ‚Äî `client.create_prompt(name, content="...", environments=["staging"])` and `client.create_chat_prompt(...)` accept `environments` directly
- **TypeScript parity** ‚Äî `setPromptEnvironments`, `getPrompt({ environment })`, and `createPrompt({ environments })` mirror the Python API
- **Sequential version numbers** ‚Äî prompt versions now show as `v1`, `v2`, `v3` in the UI and API instead of raw commit hashes
- **Environment badges everywhere** ‚Äî assigned environments appear next to every version reference in the prompt library, history timeline, diff view, and Playground
- **Terminology update** ‚Äî "commit" has been replaced with "version" throughout the prompt UI

```python
# Tag a version at creation time
prompt = client.create_prompt("system-prompt", content="...", environments=["staging"])

# Retrieve by environment ‚Äî no hard-coded version number needed
production_prompt = client.get_prompt("system-prompt", environment="production")

# Promote a specific version to production
client.set_prompt_environments("system-prompt", ["production"], version="v3")
```

## Filter Chips for the Logs View

The Traces, Spans, and Threads tabs in the Logs view now have a chip-based filter bar. There are six chip types ‚Äî single-select, pseudo-search, boolean, numeric, time range, and query-builder ‚Äî each with a dedicated popover. Chips can be pinned to the bar so frequently-used filters stay one click away, and filter state persists in the URL and is remembered per project in local storage. Clicking a tag value in any trace row adds it to the Tags chip directly, making it fast to drill into a specific label without opening the filter panel manually.

## Bug Fixes & Improvements

- **Test suite assertions: sub-span inspection** ‚Äî the evaluator LLM can now issue `get_trace_spans` and `read` tool calls to inspect intermediate spans during evaluation, enabling correctness checks about tool usage, model selection, and per-span errors inside complex agents
- **Google ADK integration: images render in trace attachments** ‚Äî URL-safe base64 image data sent by ADK is automatically normalized to standard base64; PNG, JPEG, GIF, and WebP attachments all render correctly
- **Optimization trials page: all constituent experiments shown** ‚Äî experiments belonging to multi-project optimizations are now visible from the trials page regardless of which project the user is currently viewing the optimization from
- **Error rate KPI: now shows a percentage** ‚Äî the error rate dashboard card was displaying a raw event count; it now shows the rate as a percentage
- **Annotation queue: trace logs shown inline** ‚Äî trace log entries are rendered inline on the annotation queue page instead of requiring navigation away
- **Online evaluation rules: ClassCastException resolved** ‚Äî thread-level rules that include filters no longer throw a `ClassCastException` under certain configurations
- **Attachments: data URI prefix handled** ‚Äî base64 attachment payloads that include a `data:<type>;base64,` prefix are now stripped correctly in both the SDK and the frontend
- **SDK: built-in environment colors preserved** ‚Äî workspace environments with reserved names retain their designated color after updates or syncs
- **`opik migrate`: skipped items reported clearly** ‚Äî the migration command now reports each skipped item with its reason, count, and sample source IDs, and exits with code 1 so CI pipelines detect incomplete migrations
- **Qianfan integration documentation** ‚Äî the Qianfan LLM provider integration now has a dedicated documentation page
