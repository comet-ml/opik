# `_release-gate/` — dev-authored release-gate tests

Tests here are generated by the `/explore-feature` skill, one per PR/feature, to gate that
change's own release. They are **not** part of QA's curated `@t1-smoke` / `@t2-cuj` /
`@t3-nightly` suite — they run as their own job and are triaged after they've gated a release.

The authoring rules (path, tags, stamp, deployment-agnostic requirements) live in
`.agents/skills/explore-feature/release-gate-contract.md` — that's the source of truth. This file
is the dev/QA-facing summary of the lifecycle; if the two disagree, the contract wins.

## How a test gets here

`/explore-feature` reads a ticket + changed code, writes a happy-path `test-plan.md` (posted to
Jira), and commits one staging-ready spec here:

- Path: `<lead-ticket>.spec.ts`
- Tags: `@release-gate` + `@release-gate:<version>` + a feature tag (`@datasets`, …)
- `<version>` is read from `origin/main:version.txt` at authoring time — the release this PR targets.

Run them with: `npm run test:release-gate`

## Lifecycle (folder = state)

| Location | Meaning |
|---|---|
| `_release-gate/<lead>.spec.ts` | **Active.** Gates on staging if `stamp ≤ version-being-cut` and not yet shipped-green. |
| `_release-gate/shipped/` | Gated a shipped release; inert. QA's triage queue. |
| `tests/<feature>/` | QA promoted it into the permanent regression suite. |

The CI moves (into `shipped/`, then age-expiry after ~14 days) are separate from this skill and
are not wired yet. The `/explore-feature` skill only ever **creates/appends** at this top level;
it never moves or deletes.

## Selection is a floor, not exact-match

The staging gate runs every test where `@release-gate:<version>` ≤ the version being cut and that
hasn't yet passed in a shipped release. A cancelled/skipped release therefore never orphans a
test — the next release that actually ships picks it up.
